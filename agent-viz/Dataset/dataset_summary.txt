Important format notes:
- Tool calls are often embedded inside assistant message.content as items with type="toolCall"
- Tool outputs appear as message.role="toolResult"
- Successful pattern often looks like:
  user -> assistant(with embedded toolCall) -> toolResult(success) -> assistant(final response)
- Failure pattern often looks like:
  user -> assistant(with embedded toolCall) -> toolResult(error) -> assistant(explanation)
- Retry pattern may show repeated embedded toolCall + toolResult pairs before the final assistant response

How this relates to the project:
- These files should be treated as the real OpenClaw trace dataset for the project handoff.
- Synthetic traces were used earlier to bootstrap the project pipeline and visual flow.
- These raw OpenClaw session traces are now the source of truth for real trajectory analysis.
- Higher-level failure categories such as tool misuse, early quit, reasoning gaps, or missing steps may need to be derived during ingestion, annotation, or visualization, because they are not always explicitly pre-labeled in the raw trace files.

Purpose:
- Use these files for step-by-step trajectory inspection
- Compare successful vs failed runs
- Analyze where failures emerge
- Support debugging and post-hoc human understanding of agent behavior
